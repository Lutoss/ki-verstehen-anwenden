Vier Dinge, mehr nicht. Kein bestimmtes Produkt und kein Bezahlplan sind Pflicht.

Ein kompatibles KI-Arbeitswerkzeug

Am bequemsten ist ein Agent, der deinen lokalen Projektordner lesen und schreiben kann. Beispiele sind Claude Desktop/Cowork, Claude Code oder Codex. Die genauen Schalter unterscheiden sich.

Ein normaler Projektordner

Lege einen Ordner für dein Vorhaben an. Darin liegen später Markdown-Tickets, Zieldokument, Entscheidungen und Ergebnisse. Eine Versionsverwaltung ist nicht nötig.

Skills oder Copy-Paste

Kann dein Werkzeug Skills installieren, nutzt du die sechs Kern-Skills direkt. Sonst funktionieren die sichtbaren deutschen Aufträge im normalen Chat; du pflegst die lokalen Dateien dann selbst.

Kosten sind produktabhängig

Der Kurs verlangt kein Abo. Ordnerzugriff und lange Agentenläufe können je nach Produkt einen Bezahlplan brauchen. Ohne diese Funktionen bleibt der vollständige Copy-Paste-Weg nutzbar. Stand 11.07.2026: vor einem Abo aktuell prüfen.

Und die Sprache?

Du schreibst ganz normal Deutsch, die KI antwortet Deutsch, auch wenn ihre Anleitung Englisch ist.

Konvention für lokale Umsetzungstickets

Status ist die Triage-Rolle (ready-for-agent, needs-info usw.); State ist der Lebenszyklus (open oder closed). Ein Blocker gilt erst als gelöst, wenn das abhängige Ticket State: closed trägt. Fehlen Angaben, bleibt es State: open, bekommt Status: needs-info und nennt den Grund. Erzeugt der unveränderte Original-Skill lokale Tickets noch ohne State-Zeile, ergänzt du einmalig State: open, bevor Phase 3 beginnt. Wayfinder-Klärungstickets verwenden zusätzlich ihren eigenen Ablauf mit claimed/resolved; diese Wörter nicht mit dem Umsetzungsticket-Lebenszyklus vermischen.

# Setup Matt Pocock's Skills

Scaffold the per-repo configuration that the engineering skills assume:

- **Issue tracker** — where issues live (GitHub by default; local markdown is also supported out of the box)
- **Triage labels** — the strings used for the five canonical triage roles
- **Domain docs** — where `CONTEXT.md` and ADRs live, and the consumer rules for reading them

This is a prompt-driven skill, not a deterministic script. Explore, present what you found, confirm with the user, then write.

## Process

### 1. Explore

Look at the current repo to understand its starting state. Read whatever exists; don't assume:

- `git remote -v` and `.git/config` — is this a GitHub repo? Which one?
- `AGENTS.md` and `CLAUDE.md` at the repo root — does either exist? Is there already an `## Agent skills` section in either?
- `CONTEXT.md` and `CONTEXT-MAP.md` at the repo root
- `docs/adr/` and any `src/*/docs/adr/` directories
- `docs/agents/` — does this skill's prior output already exist?
- `.scratch/` — sign that a local-markdown issue tracker convention is already in use
- Is the `triage` skill installed? (a `triage` skill folder alongside this one, or `triage` in your available skills.) This decides whether Section B runs at all.
- Monorepo signals — a `pnpm-workspace.yaml`, a `workspaces` field in `package.json`, or a populated `packages/*` with its own `src/`. Present only in a genuinely large multi-package repo; their absence means single-context, which is almost every repo.

### 2. Present findings and ask

Summarise what's present and what's missing. Then take the sections in order — one section, one answer, then the next.

Lead each section with the recommended answer so the user can accept it in a word. Give a one-line explainer only when the choice genuinely branches; skip the section entirely when exploration already settled it (Section B when `triage` isn't installed, Section C when there's no monorepo).

**Section A — Issue tracker.**

> Explainer: The "issue tracker" is where issues live for this repo. Skills like `to-tickets`, `triage`, `to-spec`, and `qa` read from and write to it — they need to know whether to call `gh issue create`, write a markdown file under `.scratch/`, or follow some other workflow you describe. Pick the place you actually track work for this repo.

Default posture: these skills were designed for GitHub. If a `git remote` points at GitHub, propose that. If a `git remote` points at GitLab (`gitlab.com` or a self-hosted host), propose GitLab. Otherwise (or if the user prefers), offer:

- **GitHub** — issues live in the repo's GitHub Issues (uses the `gh` CLI)
- **GitLab** — issues live in the repo's GitLab Issues (uses the [`glab`](https://gitlab.com/gitlab-org/cli) CLI)
- **Local markdown** — issues live as files under `.scratch/<feature>/` in this repo (good for solo projects or repos without a remote)
- **Other** (Jira, Linear, etc.) — ask the user to describe the workflow in one paragraph; the skill will record it as freeform prose

Record the choice in `docs/agents/issue-tracker.md`. The GitHub and GitLab templates carry a "PRs as a request surface" flag, defaulted **off** — leave it off and don't raise it; a user who wants external PRs in the triage queue can flip the flag in the file later.

**Section B — Triage label vocabulary.** Skip this section entirely if the `triage` skill isn't installed (exploration told you) — an uninstalled skill needs no labels.

If it is installed, ask exactly one question:

> Do you want to keep the default triage labels? (recommended: **yes**)

The defaults are the five canonical roles, each label string equal to its name: `needs-triage`, `needs-info`, `ready-for-agent`, `ready-for-human`, `wontfix`. On **yes**, write them as-is. Only if the user says no — usually because their tracker already uses other names (e.g. `bug:triage` for `needs-triage`) — collect the overrides so `triage` applies existing labels instead of creating duplicates.

**Section C — Domain docs.** Default to **single-context** — one `CONTEXT.md` + `docs/adr/` at the repo root. This fits almost every repo; write it without asking.

Offer **multi-context** — a root `CONTEXT-MAP.md` pointing to per-context `CONTEXT.md` files — only when exploration found monorepo signals. Then confirm which layout they want.

### 3. Confirm and edit

Show the user a draft of:

- The `## Agent skills` block to add to whichever of `CLAUDE.md` / `AGENTS.md` is being edited (see step 4 for selection rules)
- The contents of `docs/agents/issue-tracker.md`, `docs/agents/domain.md`, and `docs/agents/triage-labels.md` (the last only when `triage` is installed)

Let them edit before writing.

### 4. Write

**Pick the file to edit:**

- If `CLAUDE.md` exists, edit it.

A loose idea has arrived — too big for one agent session, and wrapped in fog: the way from here to the **destination** isn't visible yet. Wayfinding is about finding that way, not charging at the destination. This skill charts the way as a **shared map** on the repo's issue tracker, then works its tickets one at a time until the route is clear.

The destination varies per effort, and naming it is the first act of charting — it shapes every ticket. It might be a spec to hand off and iterate on, a decision to lock before planning starts, or a change made in place like a data-structure migration. The map is domain-agnostic — engineering work, course content, whatever fits the shape.

## Plan, don't do

Wayfinder is **planning** by default: each ticket resolves a decision, and the map is done when the way is clear — nothing left to decide before someone goes and does the thing. The pull to just do the work is usually the signal you've reached the edge of the map and it's time to hand off. An effort can override this in its **Notes** — carrying execution into the map itself — but absent that, produce decisions, not deliverables.

## Refer by name

Every map and ticket is an issue, so it has a **name** — its title. In everything the human reads — narration, the map's Decisions-so-far — refer to it by that name, never by a bare id, number, or slug. A wall of `#42, #43, #44` is illegible; names read at a glance. The id and URL don't vanish — a name wraps its link — but they ride *inside* the name, never stand in for it.

## The Map

The map is a single issue on this repo's issue tracker, labelled `wayfinder:map` — the canonical artifact. Its tickets are child issues of the map.

The map is an **index**, not a store. It lists the decisions made and points at the tickets that hold their detail; a decision lives in exactly one place — its ticket — so the map never restates it, only gists it and links.

**Where the map, its child tickets, blocking, and frontier queries physically live is tracker-specific.** The issue tracker should have been provided to you — run `/setup-matt-pocock-skills` if not. Consult the tracker doc's "Wayfinding operations" section for how _this_ repo expresses them. If no tracker has been provided, default to the local-markdown tracker.

### The map body

The whole map at low resolution, loaded once per session. Open tickets are **not** listed — they are open child issues, found by query.

```markdown
## Destination

<what reaching the end of this map looks like — the spec, decision, or change this effort is finding its way to. One or two lines; every session orients to it before choosing a ticket.>

## Notes

<domain; skills every session should consult; standing preferences for this effort>

## Decisions so far

<!-- the index — one line per closed ticket: enough to judge relevance, then zoom the link for the detail the ticket holds -->

- [<closed ticket title>](link) — <one-line gist of the answer>

## Not yet specified

<!-- see "Fog of war": in-scope fog you can't ticket yet; graduates as the frontier advances -->

## Out of scope

<!-- see "Out of scope": work ruled beyond the destination; closed, never graduates -->
```

### Tickets

Each ticket is a **child issue** of the map; the tracker's issue id is its identity. Its body is the question, sized to one 100K token agent session:

```markdown
## Question

<the decision or investigation this ticket resolves>
```

Each ticket carries a `wayfinder:<type>` label — one of `research`, `prototype`, `grilling`, `task` (see [Ticket Types](#ticket-types)).

A session **claims** a ticket by assigning it to the dev driving the map, **first**, before any work, so concurrent sessions skip it. That assignee _is_ the claim: an open, unassigned ticket is unclaimed.

Blocking uses the tracker's **native** dependency relationship — essential because it renders the frontier _visually_ in the tracker's own UI, so the human sees what's takeable without opening the map. Only a tracker that lacks native blocking falls back to a body convention. A ticket is **unblocked** when every ticket blocking it is closed; the **frontier** is the open, unblocked, unclaimed children — the edge of the known.

The answer isn't part of the body — it's recorded on resolution (see [Work through the map](#work-through-the-map)). Assets created while resolving a ticket are linked from the issue, not pasted in.

## Ticket Types

Every ticket is either **HITL** — human in the loop, worked *with* a human who speaks for themselves — or **AFK**, driven by the agent alone. A HITL ticket only resolves through that live exchange; the agent never stands in for the human's side of it (a grilling agent that answers its own questions has broken this).

- **Research** (AFK): Reading documentation, third-party APIs, or local resources like knowledge bases. Creates a markdown summary as a linked asset. Use when knowledge outside the current working directory is required.
- **Prototype** (HITL): Raise the fidelity of the discussion by making a cheap, rough, concrete artifact to react to — an outline, a rough take, a stub, or UI/logic code via the /prototype skill. Links the prototype as an asset. Use when "how should it look" or "how should it behave" is the key question.
- **Grilling** (HITL): Conversation via the /grilling and /domain-modeling skills, one question at a time. The default case.
- **Task** (HITL or AFK): Manual work that must happen before a *decision* can be made — nothing to decide, prototype, or research, but the discussion is blocked until it's done. Signing up for a service so its API can be judged, provisioning access, moving data so its shape can be seen. This is the one type that *does* rather than decides — and it earns its place by unblocking a decision, not by delivering the destination. The agent drives it alone where it can (AFK); otherwise it hands the human a precise checklist (HITL). Resolved when the work is done; the answer records what was done and any resulting facts (credentials location, new URLs, row counts) later tickets depend on.

## Fog of war

The map is _deliberately_ incomplete: don't chart what you can't yet see. Beyond the live tickets lies the **fog of war** — the dim view of decisions and investigations you can tell are coming but can't yet pin down, because they hang on questions still open. Resolving a ticket clears the fog ahead of it, graduating whatever's now specifiable into fresh tickets — one at a time, until the way to the destination is clear and no tickets remain.

The map's **Not yet specified** section is where that dim view is written down: the suspected question, the area to revisit later. It's the undiscovered frontier _toward_ the destination — everything here is in scope, just not sharp enough to ticket. Write as loosely or as fully as the view allows; it doubles as a signpost for collaborators reading where the effort is headed.

**Fog or ticket?** The test is whether you can state the question precisely now — _not_ whether you can answer it now.

- **Ticket when** the question is already sharp — even if it's blocked and you can't act on it yet.
- **Not yet specified when** you can't yet phrase it that sharply. Don't pre-slice the fog into ticket-sized pieces: it's coarser than a ticket, and one patch may graduate into several tickets, or none, once the frontier reaches it.

**Not yet specified** excludes what's already decided (Decisions so far), what's already a live ticket, and what's out of scope (the next section).

## Out of scope

Fog only ever gathers _toward_ the destination. The destination fixes the scope, so work beyond it is **out of scope** — it isn't fog, and it doesn't belong in **Not yet specified**. It gets its own **Out of scope** section on the map: work you've consciously ruled out of _this_ effort. Scope, not sharpness, lands it here.

Out-of-scope work never graduates — the frontier stops at the destination — so it returns only if the destination is redrawn, and then as a fresh effort, not a resumption.

Ruling something out of scope is a scoping act, not a step on the route. When a ticket that already exists turns out to sit past the destination — mis-scoped in while charting, or exposed by a resolution — **close it** (a closed ticket is unambiguously off the frontier) and leave one line in the **Out of scope** section: the gist plus why it's out of scope, linking the closed ticket. It stays out of **Decisions so far**, which records the route actually walked — a scope boundary isn't a step on it.

Two modes. Either way, **never resolve more than one ticket per session.**

### Chart the map

User invokes with a loose idea.

1. **Name the destination.** Run a `/grilling` and `/domain-modeling` session to pin down what this map is finding its way to — the spec, decision, or change. The destination fixes the scope, so it's settled first.
2. **Map the frontier.** Grill again, **breadth-first** this time: fan out across the whole space rather than deep on any one thread, surfacing the open decisions and the first steps takeable now. **If this surfaces no fog** — the way to the destination is already clear, the whole journey small enough for one session — you don't need a map. Stop and ask the user how they'd like to proceed.
3. **Create the map** (label `wayfinder:map`): Destination and Notes filled in, Decisions-so-far empty, the fog sketched into **Not yet specified**.
4. **Create the tickets you can specify now** as child issues of the map — then wire blocking edges in a **second pass** (issues need ids before they can reference each other). Wiring sorts them into the frontier and the blocked; everything you can't yet specify stays in the fog — the **Not yet specified** section.
5. Stop — charting the map is one session's work; do not also resolve tickets.

### Work through the map

User invokes with a map (URL or number). A ticket is **optional** — without one, you pick the next decision, not the user.

1. Load the **map** — the low-res view, not every ticket body.
2. Choose the ticket. If the user named one, use it. Otherwise take the first frontier ticket in order. **Claim it**: assign it to yourself before any work.
3. Resolve it — **zoom as needed**: fetch the full body of any related or closed ticket on demand; invoke the skills the `## Notes` block names. If in doubt, use `/grilling` and `/domain-modeling`.
4. Record the resolution: post the answer as a **resolution comment**, **close** the issue, and **append a context pointer** to the map's Decisions-so-far.
5. Add newly-surfaced tickets (create-then-wire); graduate any fog the answer has made specifiable, clearing each graduated patch from **Not yet specified** so it lives only as its new ticket. If the answer reveals a ticket — this one or another — sits beyond the destination, **rule it out of scope** rather than resolving it on the route. If the decision invalidates other parts of the map, update or delete those tickets.

The user may run unblocked tickets in parallel, so expect other sessions to be editing the tracker concurrently.

Eine lose Idee ist angekommen — zu groß für eine Agenten-Sitzung und in Nebel gehüllt: Der Weg von hier zum **Ziel** ("Destination") ist noch nicht sichtbar. Wegfinden heißt: diesen Weg finden, nicht aufs Ziel losstürmen. Dieser Skill zeichnet den Weg als **gemeinsame Landkarte** in der Aufgaben-Ablage des Projekts und arbeitet ihre Tickets dann eines pro Sitzung ab, bis die Route klar ist.

Das Ziel ist bei jedem Vorhaben anders, und es zu benennen ist der erste Akt des Kartenzeichnens — es prägt jedes Ticket. Es kann eine Spec sein, die übergeben und weiterentwickelt wird, eine Entscheidung, die vor der Planung feststehen muss, oder eine Änderung, die direkt umgesetzt wird. Die Karte ist themen-offen — Software, Kursinhalte, was auch immer diese Form hat.

## Planen, nicht machen

Wayfinder ist standardmäßig **Planung**: Jedes Ticket löst eine Entscheidung, und die Karte ist fertig, wenn der Weg klar ist — nichts mehr zu entscheiden, bevor jemand losgeht und die Sache baut. Der Drang, die Arbeit einfach selbst zu erledigen, ist meist das Signal, dass du den Rand der Karte erreicht hast und die Übergabe ansteht. Ein Vorhaben kann das in seinen **Notizen** ausdrücklich anders regeln (dann gehört auch Umsetzung auf die Karte) — ohne so eine Regel gilt: Entscheidungen liefern, keine Ergebnisse.

## Beim Namen nennen

Jede Karte und jedes Ticket ist ein Eintrag in der Ablage und hat darum einen **Namen** — seinen Titel. In allem, was der Mensch liest (Erzähltext, die "Bisherige Entscheidungen" der Karte), wird beim Namen verwiesen, nie mit einer nackten Nummer oder einem Kürzel. Eine Wand aus "#42, #43, #44" ist unlesbar; Namen versteht man auf einen Blick. Nummer und Link verschwinden nicht — der Name trägt seinen Link in sich — aber sie fahren *im* Namen mit, statt ihn zu ersetzen.

## Die Karte

Die Karte ist ein einzelner Eintrag in der Aufgaben-Ablage dieses Projekts, mit dem Etikett `wayfinder:map` — das maßgebliche Artefakt. Ihre Tickets sind Kind-Einträge der Karte.

Die Karte ist ein **Verzeichnis**, kein Lager: Sie listet die getroffenen Entscheidungen und zeigt auf die Tickets, die die Details tragen. Eine Entscheidung lebt an genau einem Ort — in ihrem Ticket. Die Karte wiederholt sie nie, sie fasst sie in einer Zeile und verlinkt.

**Wo Karte, Kind-Tickets, Blockierungen und die Frontier-Abfragen technisch liegen, hängt von der Ablage ab.** Die Aufgaben-Ablage sollte dir bekannt sein — falls nicht, führe `/setup-matt-pocock-skills` aus. Wie DIESES Projekt das ausdrückt, steht im Ablage-Dokument unter "Wayfinding operations". Ist keine Ablage eingerichtet, nimm lokale Markdown-Dateien.

### Der Karten-Text

Die ganze Karte in niedriger Auflösung, einmal pro Sitzung geladen. Offene Tickets stehen NICHT darin — sie sind offene Kind-Einträge und werden per Abfrage gefunden. Der Aufbau:

- **Ziel (Destination):** Wie das Ende dieser Karte aussieht — die Spec, Entscheidung oder Änderung, zu der dieses Vorhaben den Weg sucht. Ein, zwei Zeilen; jede Sitzung richtet sich daran aus, bevor sie ein Ticket wählt.
- **Notizen:** Themenfeld; Skills, die jede Sitzung heranziehen soll; feste Vorlieben für dieses Vorhaben.
- **Bisherige Entscheidungen:** das Verzeichnis — eine Zeile pro geschlossenem Ticket: Titel als Link plus eine Zeile Kern der Antwort. Genug, um die Relevanz zu beurteilen; die Details holt man über den Link.
- **Noch nicht greifbar:** siehe "Nebel des Krieges" — offene Fragen im Zielbereich, die sich noch nicht als Ticket formulieren lassen; rücken nach, wenn die Front vorankommt.
- **Außerhalb des Ziels:** siehe "Außerhalb des Ziels" — Arbeit, die bewusst jenseits des Ziels liegt; geschlossen, rückt nie nach.

### Tickets

Jedes Ticket ist ein **Kind-Eintrag** der Karte; seine Nummer in der Ablage ist seine Identität. Sein Text ist die Frage, bemessen auf eine Agenten-Sitzung von etwa 100.000 Token:

> **Frage:** die Entscheidung oder Untersuchung, die dieses Ticket auflöst

Jedes Ticket trägt ein Etikett `wayfinder:<typ>` — einen der Typen `research`, `prototype`, `grilling`, `task` (siehe "Ticket-Typen").

Eine Sitzung **beansprucht** ein Ticket, indem sie es der Person zuweist, die die Karte führt — und zwar **zuerst**, vor jeder Arbeit, damit parallele Sitzungen es überspringen. Die Zuweisung IST der Anspruch: Ein offenes, nicht zugewiesenes Ticket ist frei.

Blockierungen nutzen die **eingebaute** Abhängigkeits-Funktion der Ablage — wichtig, weil die Ablage die Front dann selbst sichtbar macht und der Mensch ohne Blick in die Karte sieht, was greifbar ist. Nur eine Ablage ohne eingebaute Blockierungen weicht auf eine Text-Konvention aus. Ein Ticket ist **frei**, wenn alle Tickets, die es blockieren, geschlossen sind; die **Front (Frontier)** sind die offenen, freien, unbeanspruchten Kinder — der Rand des Bekannten.

Die Antwort gehört nicht in den Ticket-Text — sie wird beim Abschluss festgehalten (siehe "Die Karte abarbeiten"). Dateien, die beim Lösen entstehen, werden vom Ticket verlinkt, nicht hineinkopiert.

## Ticket-Typen

Jedes Ticket ist entweder **HITL** — Mensch in der Schleife, gemeinsam mit einem Menschen bearbeitet, der für sich selbst spricht — oder **AFK**, vom Agenten allein getrieben. Ein HITL-Ticket wird nur durch dieses echte Gespräch gelöst; der Agent springt nie für die Menschen-Seite ein (ein Grill-Agent, der seine eigenen Fragen beantwortet, hat genau das gebrochen).

- **Research** (AFK): Doku, fremde Schnittstellen oder lokales Wissen lesen. Erzeugt eine Markdown-Zusammenfassung als verlinkte Datei. Nutzen, wenn Wissen von außerhalb des Projektordners gebraucht wird.
- **Prototype** (HITL): Das Gespräch schärfen, indem etwas Billiges, Grobes, Konkretes zum Reagieren entsteht — eine Gliederung, ein Rohentwurf, ein Gerüst oder Oberflächen-/Logik-Code über den /prototype-Skill. Der Prototyp wird als Datei verlinkt. Nutzen, wenn "Wie soll es aussehen?" oder "Wie soll es sich verhalten?" die Kernfrage ist.
- **Grilling** (HITL): Gespräch über die Skills /grilling und /domain-modeling, eine Frage nach der anderen. Der Normalfall.
- **Task** (HITL oder AFK): Handarbeit, die passieren muss, bevor eine *Entscheidung* möglich ist — nichts zu entscheiden, zu prototypen oder zu recherchieren, aber das Gespräch hängt, bis es erledigt ist. Bei einem Dienst anmelden, damit man seine Schnittstelle beurteilen kann; Zugänge einrichten; Daten umziehen, damit man ihre Form sieht. Der eine Typ, der *tut* statt entscheidet — und seinen Platz verdient er, weil er eine Entscheidung freischaltet, nicht weil er das Ziel liefert. Der Agent erledigt ihn allein, wo er kann (AFK); sonst bekommt der Mensch eine präzise Checkliste (HITL). Gelöst, wenn die Arbeit getan ist; die Antwort hält fest, was getan wurde und welche neuen Fakten (Ablageort von Zugängen, neue Adressen, Datenmengen) spätere Tickets brauchen.

## Nebel des Krieges

Die Karte ist mit Absicht unvollständig: Zeichne nicht, was du noch nicht sehen kannst. Hinter den lebenden Tickets liegt der **Nebel des Krieges** — der dunstige Blick auf Entscheidungen und Untersuchungen, die absehbar kommen, sich aber noch nicht festnageln lassen, weil sie an offenen Fragen hängen. Jedes gelöste Ticket lichtet den Nebel vor sich und lässt daraus frische Tickets nachrücken — eines nach dem anderen, bis der Weg zum Ziel klar ist und kein Ticket mehr übrig ist.

Der Karten-Abschnitt **Noch nicht greifbar** ist der Ort für diesen dunstigen Blick: die vermutete Frage, der Bereich, zu dem man zurückkommt. Er ist die unentdeckte Front IN RICHTUNG Ziel — alles darin gehört zum Vorhaben, ist nur noch nicht scharf genug für ein Ticket. Schreibe so locker oder so ausführlich, wie der Blick es erlaubt; der Abschnitt dient zugleich als Wegweiser für Mitwirkende, wohin das Vorhaben steuert.

**Nebel oder Ticket?** Der Test ist, ob du die Frage jetzt schon präzise formulieren kannst — NICHT, ob du sie jetzt beantworten kannst.

- **Ticket**, wenn die Frage schon scharf ist — auch wenn sie blockiert ist und du noch nichts tun kannst.
- **Noch nicht greifbar**, wenn du sie noch nicht so scharf fassen kannst. Zerteile den Nebel nicht vorsorglich in Ticket-Häppchen: Er ist gröber als ein Ticket, und ein Fleck kann später mehrere Tickets ergeben — oder gar keins.

"Noch nicht greifbar" enthält nicht, was schon entschieden ist (Bisherige Entscheidungen), was schon als Ticket lebt und was außerhalb des Ziels liegt (nächster Abschnitt).

## Außerhalb des Ziels

Nebel sammelt sich nur IN RICHTUNG Ziel. Das Ziel legt den Rahmen fest; Arbeit jenseits davon ist **außerhalb des Ziels** — sie ist kein Nebel und gehört nicht in "Noch nicht greifbar". Sie bekommt ihren eigenen Karten-Abschnitt: Arbeit, die du bewusst aus DIESEM Vorhaben ausgeschlossen hast. Der Rahmen entscheidet, nicht die Schärfe.

Ausgeschlossenes rückt nie nach — die Front endet am Ziel. Es kommt nur zurück, wenn das Ziel neu gezogen wird, und dann als frisches Vorhaben, nicht als Fortsetzung.

Etwas auszuschließen ist ein Rahmen-Akt, kein Schritt auf der Route. Stellt sich heraus, dass ein bestehendes Ticket jenseits des Ziels liegt — beim Zeichnen hineingerutscht oder durch eine Antwort entlarvt —, **schließe es** (ein geschlossenes Ticket ist eindeutig von der Front) und hinterlasse eine Zeile im Abschnitt "Außerhalb des Ziels": der Kern plus das Warum, mit Link auf das geschlossene Ticket. In "Bisherige Entscheidungen" gehört es nicht — dort steht die tatsächlich gegangene Route, und eine Rahmen-Grenze ist kein Schritt auf ihr.

## Aufruf

Zwei Modi. In beiden gilt: **Nie mehr als ein Ticket pro Sitzung lösen.**

### Die Karte zeichnen

Der Nutzer ruft den Skill mit einer losen Idee auf.

1. **Das Ziel benennen.** Eine /grilling- und /domain-modeling-Runde fahren, um festzunageln, wohin diese Karte den Weg sucht — die Spec, Entscheidung oder Änderung. Das Ziel legt den Rahmen fest, also wird es zuerst geklärt.
2. **Die Front kartieren.** Noch einmal grillen, diesmal **in die Breite**: über das ganze Feld fächern statt tief in einen Strang, die offenen Entscheidungen und die jetzt schon greifbaren ersten Schritte ans Licht holen. **Taucht dabei kein Nebel auf** — der Weg zum Ziel ist schon klar, die ganze Reise klein genug für eine Sitzung —, braucht es keine Karte. Anhalten und den Nutzer fragen, wie er weitermachen will.
3. **Die Karte anlegen** (Etikett `wayfinder:map`): Ziel und Notizen ausgefüllt, Bisherige Entscheidungen leer, der Nebel in "Noch nicht greifbar" skizziert.
4. **Die jetzt formulierbaren Tickets anlegen**, als Kind-Einträge der Karte — die Blockierungs-Kanten dann in einem **zweiten Durchgang** verdrahten (Einträge brauchen erst Nummern, bevor sie aufeinander verweisen können). Das Verdrahten sortiert sie in Front und Blockierte; alles noch nicht Formulierbare bleibt im Nebel — im Abschnitt "Noch nicht greifbar".
5. Aufhören — die Karte zu zeichnen ist die Arbeit einer Sitzung; löse nicht auch noch Tickets.

### Die Karte abarbeiten

Der Nutzer ruft den Skill mit einer Karte auf (Link oder Nummer). Ein Ticket zu nennen ist **optional** — ohne eines wählst du die nächste Entscheidung, nicht der Nutzer.

1. Lade die **Karte** — die niedrig aufgelöste Sicht, nicht jeden Ticket-Text.
2. Wähle das Ticket. Hat der Nutzer eines genannt, nimm das. Sonst nimm das erste Front-Ticket der Reihe nach. **Beanspruche es**: Weise es dir zu, bevor du irgendetwas tust.
3. Löse es — **zoome nach Bedarf**: Hole dir den vollen Text verwandter oder geschlossener Tickets, wenn du ihn brauchst; rufe die Skills auf, die der Notizen-Block nennt. Im Zweifel /grilling und /domain-modeling.
4. Halte die Lösung fest: die Antwort als **Abschluss-Kommentar** ans Ticket, das Ticket **schließen**, und eine Verweiszeile an "Bisherige Entscheidungen" der Karte anhängen.
5. Neu aufgetauchte Tickets anlegen (erst anlegen, dann verdrahten); Nebel, den die Antwort greifbar gemacht hat, zu Tickets befördern und die beförderten Flecken aus "Noch nicht greifbar" streichen, damit sie nur noch als Ticket leben. Zeigt die Antwort, dass ein Ticket — dieses oder ein anderes — jenseits des Ziels liegt, **schließe es als außerhalb des Ziels**, statt es auf der Route zu lösen. Entwertet die Entscheidung andere Teile der Karte, aktualisiere oder lösche die betroffenen Tickets.

Der Nutzer darf freie Tickets parallel laufen lassen — rechne also damit, dass andere Sitzungen die Ablage gleichzeitig bearbeiten.

# Domain Modeling

Actively build and sharpen the project's domain model as you design. This is the *active* discipline — challenging terms, inventing edge-case scenarios, and writing the glossary and decisions down the moment they crystallise. (Merely *reading* `CONTEXT.md` for vocabulary is not this skill — that's a one-line habit any skill can do. This skill is for when you're changing the model, not just consuming it.)

## File structure

Most repos have a single context:

```
/
├── CONTEXT.md
├── docs/
│   └── adr/
│       ├── 0001-event-sourced-orders.md
│       └── 0002-postgres-for-write-model.md
└── src/
```

If a `CONTEXT-MAP.md` exists at the root, the repo has multiple contexts. The map points to where each one lives:

```
/
├── CONTEXT-MAP.md
├── docs/
│   └── adr/                          ← system-wide decisions
├── src/
│   ├── ordering/
│   │   ├── CONTEXT.md
│   │   └── docs/adr/                 ← context-specific decisions
│   └── billing/
│       ├── CONTEXT.md
│       └── docs/adr/
```

Create files lazily — only when you have something to write. If no `CONTEXT.md` exists, create one when the first term is resolved. If no `docs/adr/` exists, create it when the first ADR is needed.

## During the session

### Challenge against the glossary

When the user uses a term that conflicts with the existing language in `CONTEXT.md`, call it out immediately. "Your glossary defines 'cancellation' as X, but you seem to mean Y — which is it?"

### Sharpen fuzzy language

When the user uses vague or overloaded terms, propose a precise canonical term. "You're saying 'account' — do you mean the Customer or the User? Those are different things."

### Discuss concrete scenarios

When domain relationships are being discussed, stress-test them with specific scenarios. Invent scenarios that probe edge cases and force the user to be precise about the boundaries between concepts.

### Cross-reference with code

When the user states how something works, check whether the code agrees. If you find a contradiction, surface it: "Your code cancels entire Orders, but you just said partial cancellation is possible — which is right?"

### Update CONTEXT.md inline

When a term is resolved, update `CONTEXT.md` right there. Don't batch these up — capture them as they happen. Use the format in [CONTEXT-FORMAT.md](./CONTEXT-FORMAT.md).

`CONTEXT.md` should be totally devoid of implementation details. Do not treat `CONTEXT.md` as a spec, a scratch pad, or a repository for implementation decisions. It is a glossary and nothing else.

### Offer ADRs sparingly

Only offer to create an ADR when all three are true:

1. **Hard to reverse** — the cost of changing your mind later is meaningful
2. **Surprising without context** — a future reader will wonder "why did they do it this way?"
3. **The result of a real trade-off** — there were genuine alternatives and you picked one for specific reasons

If any of the three is missing, skip the ADR. Use the format in [ADR-FORMAT.md](./ADR-FORMAT.md).

# Domain Modeling

Baue und schärfe das Domänenmodell aktiv, während ihr entwerft. Das ist eine *aktive* Disziplin: Begriffe hinterfragen, Grenzfälle erfinden und Glossar sowie Entscheidungen genau dann aufschreiben, wenn sie klar werden. (`CONTEXT.md` nur wegen der Begriffe zu *lesen* ist noch nicht dieser Skill; das ist eine kurze Gewohnheit für jeden Skill. Hier geht es darum, das Modell zu verändern, nicht nur zu benutzen.)

## Dateistruktur

Die meisten Projektordner haben einen einzigen Kontext:

```
/
├── CONTEXT.md
├── docs/
│   └── adr/
│       ├── 0001-event-sourced-orders.md
│       └── 0002-postgres-for-write-model.md
└── src/
```

Liegt im Hauptordner eine `CONTEXT-MAP.md`, hat das Projekt mehrere Kontexte. Die Karte zeigt, wo jeder davon liegt:

```
/
├── CONTEXT-MAP.md
├── docs/
│   └── adr/                          ← projektweite Entscheidungen
├── src/
│   ├── ordering/
│   │   ├── CONTEXT.md
│   │   └── docs/adr/                 ← Entscheidungen dieses Kontexts
│   └── billing/
│       ├── CONTEXT.md
│       └── docs/adr/
```

Lege Dateien erst dann an, wenn wirklich etwas hineingehört. Gibt es noch keine `CONTEXT.md`, entsteht sie beim ersten geklärten Begriff. Gibt es noch kein `docs/adr/`, entsteht es bei der ersten nötigen ADR.

## Während der Sitzung

### Gegen das Glossar prüfen

Verwendet der Nutzer einen Begriff anders als in `CONTEXT.md`, sprich den Widerspruch sofort an: "Im Glossar bedeutet 'Stornierung' X, hier scheinst du Y zu meinen. Was gilt?"

### Unscharfe Sprache präzisieren

Bei vagen oder doppeldeutigen Begriffen schlägst du einen genauen kanonischen Begriff vor: "Mit 'Konto' — meinst du den Kunden oder den Benutzer? Das sind zwei verschiedene Dinge."

### Konkrete Szenarien besprechen

Wenn Beziehungen zwischen Fachbegriffen besprochen werden, teste sie mit konkreten Grenzfällen. Erfinde Szenarien, die die Grenzen zwischen den Begriffen sichtbar machen und den Nutzer zu Präzision zwingen.

### Mit dem Projektstand abgleichen

Wenn der Nutzer erklärt, wie etwas funktioniert, prüfe, ob die vorhandenen Projektdateien dazu passen. Bei einem Widerspruch sprich ihn an: "Dein Projekt storniert ganze Bestellungen, aber du sagst gerade, Teilstornierungen seien möglich. Was ist richtig?"

### CONTEXT.md sofort aktualisieren

Sobald ein Begriff geklärt ist, aktualisiere `CONTEXT.md` an Ort und Stelle. Sammle die Änderungen nicht bis zum Schluss. Nutze das Format aus [CONTEXT-FORMAT.md](./CONTEXT-FORMAT.md).

`CONTEXT.md` enthält keinerlei Umsetzungsdetails. Behandle sie nicht als Spec, Notizzettel oder Ablage für technische Entscheidungen. Sie ist ausschließlich das Glossar.

### ADRs sparsam anbieten

Biete eine ADR nur an, wenn alle drei Punkte gelten:

1. **Schwer rückgängig zu machen** — ein späterer Wechsel wäre spürbar teuer
2. **Ohne Kontext überraschend** — künftige Leser würden sich fragen, warum es so entschieden wurde
3. **Ergebnis eines echten Abwägens** — es gab ernsthafte Alternativen und ihr habt aus konkreten Gründen gewählt

Fehlt einer der drei Punkte, verzichte auf die ADR. Nutze das Format aus [ADR-FORMAT.md](./ADR-FORMAT.md).

# To Tickets

Break a plan, spec, or conversation into a set of **tickets** — tracer-bullet vertical slices, each declaring the tickets that **block** it.

The issue tracker and triage label vocabulary should have been provided to you — run `/setup-matt-pocock-skills` if not.

## Process

### 1. Gather context

Work from whatever is already in the conversation context. If the user passes a reference (a spec path, an issue number or URL) as an argument, fetch it and read its full body and comments.

### 2. Explore the codebase (optional)

If you have not already explored the codebase, do so to understand the current state of the code. Ticket titles and descriptions should use the project's domain glossary vocabulary, and respect ADRs in the area you're touching.

Look for opportunities to prefactor the code to make the implementation easier. "Make the change easy, then make the easy change."

### 3. Draft vertical slices

Break the work into **tracer bullet** tickets.

<vertical-slice-rules>

- Each slice cuts a narrow but COMPLETE path through every layer (schema, API, UI, tests) — vertical, NOT a horizontal slice of one layer
- A completed slice is demoable or verifiable on its own
- Each slice is sized to fit in a single fresh context window
- Any prefactoring should be done first

</vertical-slice-rules>

Give each ticket its **blocking edges** — the other tickets that must complete before it can start. A ticket with no blockers can start immediately.

**Wide refactors are the exception to vertical slicing.** A **wide refactor** is one mechanical change — rename a column, retype a shared symbol — whose **blast radius** fans across the whole codebase, so a single edit breaks thousands of call sites at once and no vertical slice can land green. Don't force it into a tracer bullet; sequence it as **expand–contract**. First expand: add the new form beside the old so nothing breaks. Then migrate the call sites over in batches sized by blast radius (per package, per directory), each batch its own ticket blocked by the expand, keeping CI green batch to batch because the old form still exists. Finally contract: delete the old form once no caller remains, in a ticket blocked by every migrate batch. When even the batches can't stay green alone, keep the sequence but let them share an integration branch that all block a final integrate-and-verify ticket — green is promised only there.

### 4. Quiz the user

Present the proposed breakdown as a numbered list. For each ticket, show:

- **Title**: short descriptive name
- **Blocked by**: which other tickets (if any) must complete first
- **What it delivers**: the end-to-end behaviour this ticket makes work

Ask the user:

- Does the granularity feel right? (too coarse / too fine)
- Are the blocking edges correct — does each ticket only depend on tickets that genuinely gate it?
- Should any tickets be merged or split further?

Iterate until the user approves the breakdown.

### 5. Publish the tickets to the configured tracker

Publish the approved tickets. **How** depends on the tracker `/setup-matt-pocock-skills` configured — the tickets are the same either way, only the shape of the blocking edges changes:

- **Local files** → write one file per ticket under `.scratch/<feature-slug>/issues/<NN>-<slug>.md`, numbered from `01` in dependency order (blockers first). Each file's "Blocked by" lists the numbers/titles it depends on. Use the per-ticket file template below — one ticket per file, never a single combined file.
- **A real issue tracker (GitHub, Linear, …)** → publish one issue per ticket in dependency order (blockers first) so each ticket's blocking edges can reference real identifiers. Use the platform's native blocking / sub-issue relationship where it has one; otherwise set each ticket's "Blocked by" to the blocking issues. Apply the `ready-for-agent` triage label unless instructed otherwise — the tickets are agent-grabbable by construction.

Work the **frontier**: any ticket whose blockers are all done. For a purely linear chain that means top to bottom.

Do NOT close or modify any parent issue.

Trim to the decision-rich parts — not a working demo, just the important bits.

# Zu Tickets

Zerlege einen Plan, eine Spec oder ein Gespräch in einen Satz **Tickets** — vertikale Scheiben nach dem Leuchtspur-Prinzip (tracer bullets), von denen jedes die Tickets nennt, die es **blockieren**.

Aufgaben-Ablage und Etiketten-Vokabular sollten dir vorliegen — falls nicht, führe `/setup-matt-pocock-skills` aus.

## Ablauf

### 1. Kontext sammeln

Arbeite mit dem, was schon im Gespräch liegt. Nennt der Nutzer eine Referenz (einen Spec-Pfad, eine Ticket-Nummer oder einen Link), hole sie und lies sie vollständig, samt Kommentaren.

### 2. Projektordner erkunden (optional)

Falls noch nicht geschehen, erkunde den Projektordner, um den aktuellen Stand zu verstehen. Titel und Beschreibungen der Tickets nutzen die Begriffe aus dem Projekt-Glossar und respektieren festgehaltene Entscheidungen (ADRs) in dem Bereich.

Suche nach Gelegenheiten, vorab aufzuräumen, damit die Umsetzung leichter wird. "Mach die Änderung leicht — dann mach die leichte Änderung."

### 3. Vertikale Scheiben entwerfen

Zerlege die Arbeit in **Leuchtspur**-Tickets.

<scheiben-regeln>

- Jede Scheibe schneidet einen schmalen, aber KOMPLETTEN Weg durch alle Ebenen — vertikal, KEINE horizontale Schicht einer einzelnen Ebene
- Eine fertige Scheibe ist für sich vorzeigbar oder prüfbar
- Jede Scheibe ist so bemessen, dass sie in ein einzelnes frisches Kontextfenster passt
- Aufräumarbeiten kommen zuerst

</scheiben-regeln>

Gib jedem Ticket seine **Blockierungs-Kanten** — die anderen Tickets, die fertig sein müssen, bevor es starten kann. Ein Ticket ohne Blocker kann sofort starten.

**Breite Umbauten sind die Ausnahme vom vertikalen Schneiden.** Ein **breiter Umbau** ist eine einzelne mechanische Änderung — eine Spalte umbenennen, einen überall benutzten Baustein umtypen —, deren **Wirkungsradius** sich über das ganze Projekt auffächert: Ein einzelner Eingriff bricht tausend Stellen auf einmal, keine vertikale Scheibe kann sauber landen. Zwänge so etwas nicht in eine Leuchtspur, sondern plane es als **Erweitern–Zusammenziehen** (expand–contract). Erst erweitern: die neue Form NEBEN die alte stellen, damit nichts bricht. Dann die Verwendungsstellen in Portionen umziehen (pro Paket, pro Ordner — bemessen am Wirkungsradius), jede Portion ihr eigenes Ticket, blockiert vom Erweitern-Ticket; alles bleibt dabei lauffähig, weil die alte Form noch existiert. Zuletzt zusammenziehen: die alte Form löschen, sobald niemand sie mehr benutzt — in einem Ticket, das von allen Umzugs-Portionen blockiert wird. Können selbst die Portionen nicht einzeln sauber landen, behalte die Reihenfolge, aber lass sie sich einen gemeinsamen Arbeitszweig teilen, der in ein letztes Zusammenführen-und-prüfen-Ticket mündet — "läuft sauber" wird nur dort versprochen.

### 4. Den Nutzer befragen

Lege die vorgeschlagene Zerlegung als nummerierte Liste vor. Zeige je Ticket:

- **Titel**: kurzer, sprechender Name
- **Blockiert von**: welche anderen Tickets (falls vorhanden) vorher fertig sein müssen
- **Was es liefert**: das Verhalten von Anfang bis Ende, das dieses Ticket zum Laufen bringt

Frage den Nutzer:

- Passt die Körnung? (zu grob / zu fein)
- Stimmen die Blockierungs-Kanten — hängt jedes Ticket nur von Tickets ab, die es wirklich aufhalten?
- Sollen Tickets zusammengelegt oder weiter geteilt werden?

Wiederhole, bis der Nutzer die Zerlegung freigibt.

### 5. Tickets in der eingerichteten Ablage anlegen

Lege die freigegebenen Tickets an. **Wie**, hängt von der Ablage ab, die `/setup-matt-pocock-skills` eingerichtet hat — die Tickets sind dieselben, nur die Form der Blockierungs-Kanten ändert sich:

- **Lokale Dateien** → eine Datei pro Ticket unter `.scratch/<vorhaben>/issues/<NN>-<kürzel>.md`, ab `01` in Abhängigkeits-Reihenfolge nummeriert (Blocker zuerst). Das Feld "Blockiert von" jeder Datei nennt die Nummern/Titel, von denen sie abhängt. Nutze die Datei-Vorlage unten — ein Ticket pro Datei, nie eine einzige Sammeldatei.
- **Eine echte Aufgaben-Ablage (GitHub, Linear, …)** → ein Eintrag pro Ticket, in Abhängigkeits-Reihenfolge (Blocker zuerst), damit die Blockierungs-Kanten auf echte Nummern verweisen können. Nutze die eingebaute Blockierungs- bzw. Unteraufgaben-Beziehung der Plattform, wo es sie gibt; sonst das Feld "Blockiert von". Vergib das Etikett `ready-for-agent`, sofern nichts anderes gesagt wurde — die Tickets sind konstruktionsbedingt agenten-greifbar.

Arbeite die **Front (Frontier)**: jedes Ticket, dessen Blocker alle erledigt sind. Bei einer rein linearen Kette heißt das: von oben nach unten.

Übergeordnete Tickets werden NICHT geschlossen oder verändert.

<ticket-vorlage>

**Was gebaut wird:** das Verhalten von Anfang bis Ende, das dieses Ticket zum Laufen bringt — aus Sicht des Nutzers, keine Umsetzungsliste Ebene für Ebene.

**Blockiert von:** die Nummern/Titel der Tickets, die dieses aufhalten, oder "Keine — kann sofort starten".

**Fertig-Kriterien:** eine prüfbare Abhak-Liste (Kriterium 1, Kriterium 2, …).

</ticket-vorlage>

In beiden Formen: keine konkreten Dateipfade oder Code-Schnipsel — die veralten schnell. Ausnahme: Prototyp-Schnipsel, die eine Entscheidung präziser festhalten als Prosa; kurz die Herkunft vermerken und aufs Entscheidende kürzen.

Arbeite die Front ein Ticket nach dem anderen mit `/implement` ab und leere den Kontext zwischen den Tickets.

Ein Ticket, eine frische Sitzung, ein gespeichertes Ergebnis

Zweck: Du wählst das nächste freie lokale Markdown-Ticket und gibst einem frischen Agenten einen toolneutralen Arbeitsauftrag. Er liest Ticket, Zieldokument und Projektregeln, erstellt das Ergebnis, prüft die Fertigkriterien, speichert eine klar benannte Datei und aktualisiert Status, State und Comments nach der festgelegten Konvention. Werkstatt-Prinzip

"Ein Ticket pro frischem Kontext; fertig ist es erst mit Datei, Kriterienprüfung und ehrlichem Status."

Die Planungsroute liefert alles, was ein frischer Agent braucht. Das lokale Ticket ist die Übergabe, die Ergebnisdatei der Beleg. So bleibt der Chat kurz und der Projektordner trägt den Stand.   2. Wähle das erste freie, nicht blockierte Ticket aus der Ablage.

Prüfkriterien vorher festlegen, klein bauen und frisch prüfen ist projektneutral. Die konkreten Skills /tdd, /implement und /code-review bleiben jedoch Softwarewerkzeuge: Tests, Typechecking, Git-Diff und Commit sind echte Voraussetzungen, keine Wörter, die wir für Alltagsprojekte umdeuten.

---
name: code-review
description: Review the changes since a fixed point (commit, branch, tag, or merge-base) along two axes — Standards (does the code follow this repo's documented coding standards?) and Spec (does the code match what the originating issue/PRD asked for?). Runs both reviews in parallel sub-agents and reports them side by side. Use when the user wants to review a branch, a PR, work-in-progress changes, or asks to "review since X".
---

Two-axis review of the diff between `HEAD` and a fixed point the user supplies:

- **Standards** — does the code conform to this repo's documented coding standards?
- **Spec** — does the code faithfully implement the originating issue / PRD / spec?

Both axes run as **parallel sub-agents** so they don't pollute each other's context, then this skill aggregates their findings.

The issue tracker should have been provided to you — run `/setup-matt-pocock-skills` if `docs/agents/issue-tracker.md` is missing.

## Process

### 1. Pin the fixed point

Whatever the user said is the fixed point — a commit SHA, branch name, tag, `main`, `HEAD~5`, etc. If they didn't specify one, ask for it.

Capture the diff command once: `git diff <fixed-point>...HEAD` (three-dot, so the comparison is against the merge-base). Also note the list of commits via `git log <fixed-point>..HEAD --oneline`.

Before going further, confirm the fixed point resolves (`git rev-parse <fixed-point>`) and the diff is non-empty. A bad ref or empty diff should fail here — not inside two parallel sub-agents.

### 2. Identify the spec source

Look for the originating spec, in this order:

1. Issue references in the commit messages (`#123`, `Closes #45`, GitLab `!67`, etc.) — fetch via the workflow in `docs/agents/issue-tracker.md`.
2. A path the user passed as an argument.
3. A PRD/spec file under `docs/`, `specs/`, or `.scratch/` matching the branch name or feature.
4. If nothing is found, ask the user where the spec is. If they say there isn't one, the **Spec** sub-agent will skip and report "no spec available".

### 3. Identify the standards sources

Anything in the repo that documents how code should be written, such as `CODING_STANDARDS.md` or `CONTRIBUTING.md`.

On top of whatever the repo documents, the Standards axis always carries the **smell baseline** below — a fixed set of Fowler code smells (_Refactoring_, ch.3) that applies even when a repo documents nothing. Two rules bind it:

- **The repo overrides.** A documented repo standard always wins; where it endorses something the baseline would flag, suppress the smell.
- **Always a judgement call.** Each smell is a labelled heuristic ("possible Feature Envy"), never a hard violation — and, like any standard here, skip anything tooling already enforces.

Each smell reads *what it is* → *how to fix*; match it against the diff:

- **Mysterious Name** — a function, variable, or type whose name doesn't reveal what it does or holds. → rename it; if no honest name comes, the design's murky.
- **Duplicated Code** — the same logic shape appears in more than one hunk or file in the change. → extract the shared shape, call it from both.
- **Feature Envy** — a method that reaches into another object's data more than its own. → move the method onto the data it envies.
- **Data Clumps** — the same few fields or params keep travelling together (a type wanting to be born). → bundle them into one type, pass that.
- **Primitive Obsession** — a primitive or string standing in for a domain concept that deserves its own type. → give the concept its own small type.
- **Repeated Switches** — the same `switch`/`if`-cascade on the same type recurs across the change. → replace with polymorphism, or one map both sites share.
- **Shotgun Surgery** — one logical change forces scattered edits across many files in the diff. → gather what changes together into one module.
- **Divergent Change** — one file or module is edited for several unrelated reasons. → split so each module changes for one reason.
- **Speculative Generality** — abstraction, parameters, or hooks added for needs the spec doesn't have. → delete it; inline back until a real need shows.
- **Message Chains** — long `a.b().c().d()` navigation the caller shouldn't depend on. → hide the walk behind one method on the first object.
- **Middle Man** — a class or function that mostly just delegates onward. → cut it, call the real target direct.
- **Refused Bequest** — a subclass or implementer that ignores or overrides most of what it inherits. → drop the inheritance, use composition.

### 4. Spawn both sub-agents in parallel

Send a single message with two `Agent` tool calls. Use the `general-purpose` subagent for both.

**Standards sub-agent prompt** — include:

- The full diff command and commit list.
- The list of standards-source files you found in step 3, **plus the smell baseline from step 3** pasted in full — the sub-agent has no other access to it.
- The brief: "Report — per file/hunk where relevant — (a) every place the diff violates a documented standard: cite the standard (file + the rule); and (b) any baseline smell you spot: name it and quote the hunk. Distinguish hard violations from judgement calls — documented-standard breaches can be hard, but baseline smells are always judgement calls, and a documented repo standard overrides the baseline. Skip anything tooling enforces. Under 400 words."

**Spec sub-agent prompt** — include:

- The diff command and commit list.
- The path or fetched contents of the spec.
- The brief: "Report: (a) requirements the spec asked for that are missing or partial; (b) behaviour in the diff that wasn't asked for (scope creep); (c) requirements that look implemented but where the implementation looks wrong. Quote the spec line for each finding. Under 400 words."

If the spec is missing, skip the Spec sub-agent and note this in the final report.

### 5. Aggregate

Present the two reports under `## Standards` and `## Spec` headings, verbatim or lightly cleaned. Do **not** merge or rerank findings — the two axes are deliberately separate (see _Why two axes_).

End with a one-line summary: total findings per axis, and the worst issue _within each axis_ (if any). Don't pick a single winner across axes — that's the reranking the separation exists to prevent.

## Why two axes

A change can pass one axis and fail the other:

- Code that follows every standard but implements the wrong thing → **Standards pass, Spec fail.**
- Code that does exactly what the issue asked but breaks the project's conventions → **Spec pass, Standards fail.**

Reporting them separately stops one axis from masking the other.

## Ablauf

### 1. Den Fixpunkt festnageln

Was der Nutzer genannt hat, ist der Fixpunkt — ein Speicherpunkt, ein Zweig-Name, ein Tag, `main`, "die letzten 5 Schritte" usw. Hat er keinen genannt: fragen.

Halte den Vergleichsbefehl einmal fest (Diff gegen den gemeinsamen Abzweigpunkt) und notiere die Liste der Speicherpunkte seither.

Bevor es weitergeht: prüfen, dass der Fixpunkt existiert und der Unterschied nicht leer ist. Ein falscher Bezugspunkt oder ein leerer Diff soll HIER scheitern — nicht erst in zwei parallelen Sub-Agenten.

### 2. Die Spec-Quelle finden

Suche die ursprüngliche Aufgabenbeschreibung, in dieser Reihenfolge:

1. Ticket-Verweise in den Commit-Nachrichten (Nummern wie `#123`, "Closes #45" usw.) — über den Weg holen, der in `docs/agents/issue-tracker.md` steht.
2. Ein Pfad, den der Nutzer mitgegeben hat.
3. Eine Spec-Datei unter `docs/`, `specs/` oder `.scratch/`, die zum Zweig- oder Vorhaben-Namen passt.
4. Wird nichts gefunden: den Nutzer fragen, wo die Spec liegt. Gibt es keine, überspringt die **Spec**-Achse und meldet "keine Spec verfügbar".

### 3. Die Standard-Quellen finden

Alles im Projekt, das dokumentiert, wie Code auszusehen hat — etwa `CODING_STANDARDS.md` oder `CONTRIBUTING.md`.

Zusätzlich trägt die Standards-Achse immer die **Geruchs-Grundlinie** unten — ein fester Satz klassischer Code-Gerüche nach Fowler (_Refactoring_, Kapitel 3), der auch dann gilt, wenn das Projekt nichts dokumentiert. Zwei Regeln binden sie:

- **Das Projekt gewinnt.** Ein dokumentierter Projekt-Standard schlägt die Grundlinie; erlaubt er ausdrücklich, was die Grundlinie anmeckern würde, wird der Geruch unterdrückt.
- **Immer Ermessenssache.** Jeder Geruch ist eine benannte Vermutung ("mögliches Fremdeln"), nie ein harter Verstoß — und wie bei jedem Standard gilt: auslassen, was Prüfwerkzeuge ohnehin schon erzwingen.

Jeder Geruch liest sich *was es ist* → *wie beheben*; gleiche ihn mit den Änderungen ab:

- **Rätselhafter Name**: Eine Funktion, Variable oder ein Typ, deren Name nicht verrät, was sie tut oder enthält. → Umbenennen; fällt kein ehrlicher Name ein, ist der Entwurf unklar.
- **Doppelter Code**: Dieselbe Logik-Form taucht in der Änderung mehrfach auf. → Die gemeinsame Form herausziehen, von beiden Stellen aufrufen.
- **Fremdeln**: Eine Methode greift mehr auf fremde Daten zu als auf die eigenen. → Die Methode zu den Daten umziehen, nach denen sie schielt.
- **Daten-Grüppchen**: Dieselben Felder oder Parameter reisen immer zusammen (ein Typ, der geboren werden will). → Bündeln und das Bündel übergeben.
- **Nackte Behelfswerte**: Eine bloße Zahl oder Zeichenkette steht für einen Fachbegriff, der einen eigenen Typ verdient. → Dem Begriff einen eigenen kleinen Typ geben.
- **Wiederholte Weichen**: Dieselbe Fallunterscheidung über denselben Typ wiederholt sich in der Änderung. → Durch Polymorphie ersetzen oder eine gemeinsame Zuordnung, die beide Stellen teilen.
- **Streu-Operation**: Eine logische Änderung erzwingt verstreute Eingriffe in vielen Dateien. → Zusammenlegen, was sich gemeinsam ändert.
- **Vielzweck-Baustelle**: Eine Datei oder ein Modul wird aus mehreren, nicht verwandten Gründen geändert. → Aufteilen, bis jedes Modul nur noch einen Änderungsgrund hat.
- **Vorrats-Verallgemeinerung**: Abstraktionen, Parameter oder Haken für Bedürfnisse, die die Spec nicht hat. → Löschen; zurückbauen, bis ein echter Bedarf auftaucht.
- **Befehlsketten**: Lange Navigations-Ketten (a.b().c().d()), von denen der Aufrufer nicht abhängen sollte. → Den Weg hinter einer Methode am ersten Objekt verstecken.
- **Durchreicher**: Eine Klasse oder Funktion, die fast nur weiterdelegiert. → Streichen, das echte Ziel direkt aufrufen.
- **Ausgeschlagenes Erbe**: Eine Ableitung, die das meiste Geerbte ignoriert oder übersteuert. → Die Vererbung aufgeben, Zusammensetzung nutzen.

### 4. Beide Sub-Agenten parallel starten

Eine einzelne Nachricht mit zwei Agenten-Aufrufen senden; für beide den Allzweck-Subagenten verwenden.

**Standards-Prüfauftrag** enthält: den Vergleichsbefehl und die Liste der Speicherpunkte, die gefundenen Standard-Dateien **plus die Geruchs-Grundlinie in voller Länge** (der Sub-Agent hat sonst keinen Zugriff darauf) und den Auftrag: "Berichte — je Datei/Abschnitt, wo sinnvoll — (a) jede Stelle, an der die Änderung einen dokumentierten Standard verletzt: zitiere den Standard (Datei + Regel); und (b) jeden Grundlinien-Geruch, den du entdeckst: benenne ihn und zitiere die Stelle. Trenne harte Verstöße von Ermessensfragen — Verstöße gegen dokumentierte Standards können hart sein, Grundlinien-Gerüche sind immer Ermessenssache, und ein dokumentierter Projekt-Standard schlägt die Grundlinie. Lass aus, was Werkzeuge ohnehin erzwingen. Unter 400 Wörter."

**Spec-Prüfauftrag** enthält: den Vergleichsbefehl und die Speicherpunkt-Liste, den Pfad oder Inhalt der Spec und den Auftrag: "Berichte: (a) Verlangtes, das fehlt oder nur teilweise da ist; (b) Verhalten in der Änderung, das niemand verlangt hat (Ausuferung); (c) Verlangtes, das umgesetzt aussieht, aber falsch umgesetzt wirkt. Zitiere zu jedem Befund die Spec-Zeile. Unter 400 Wörter."

Fehlt die Spec, entfällt der Spec-Prüfauftrag; das wird im Endbericht vermerkt.

### 5. Zusammenführen

Beide Berichte unter "Standards" und "Spec" getrennt zeigen, wörtlich oder leicht geglättet. Befunde NICHT mischen und NICHT neu gewichten — die zwei Achsen sind absichtlich getrennt (siehe "Warum zwei Achsen").

Am Ende eine Zeile: Befunde je Achse und das gravierendste Problem INNERHALB jeder Achse (falls vorhanden). Keinen Gesamtsieger über die Achsen hinweg küren — genau dieses Umsortieren soll die Trennung verhindern.

## Warum zwei Achsen

Eine Änderung kann eine Achse bestehen und die andere reißen:

- Code, der jeden Standard einhält, aber das Falsche baut → **Standards ja, Spec nein.**
- Code, der genau tut, was das Ticket verlangt, aber die Gepflogenheiten des Projekts bricht → **Spec ja, Standards nein.**

Getrennt berichten verhindert, dass eine Achse die andere verdeckt.

Prüfe das beigefügte Ergebnis in zwei strikt getrennten Achsen.

1. Qualität: Ist das Ergebnis korrekt, vollständig, verständlich, nutzbar und konsistent? Folgt es den beigefügten Projektregeln? Sind konkrete Behauptungen und Quellen belegt?
2. Auftrag: Erfüllt es Ticket und Zieldokument vollständig? Nenne Fehlendes, falsch Umgesetztes und nicht bestellte Extras.

Belege jeden Befund mit einer konkreten Stelle aus Ergebnis, Regeln oder Auftrag. Vermische die Achsen nicht. Melde ausdrücklich "kein Befund", wenn eine Achse sauber ist. Nenne höchstens drei nächste Verbesserungen und nur solche, die sich aus einem wirklichen Befund ergeben. Erfinde keine Verbesserung, um die Zahl zu füllen.

Ticket und Zieldokument: [nennen oder beifügen]
Projektregeln: [nennen oder beifügen]
Software-Vertiefung · Fowlers zwölf Code-Warnzeichen
Das Geruchs-Vokabular: zwölf klassische Warnzeichen (nach Martin Fowler), in Alltagssprache
  • Rätselhafter Name (mysterious name): Der Name einer Datei, Spalte oder Funktion verrät nicht, was drinsteckt. → Ehrlich umbenennen; fällt kein ehrlicher Name ein, ist die Sache selbst unklar.
  • Doppelter Inhalt (duplicated code): Dieselbe Logik oder Information steht an zwei Stellen — und driftet auseinander. → Einmal pflegen, von beiden Stellen darauf verweisen.
  • Fremdeln (feature envy): Ein Teil interessiert sich mehr für die Daten eines anderen Teils als für die eigenen. → Dorthin umziehen, wo die Daten wohnen.
  • Daten-Grüppchen (data clumps): Dieselben Angaben reisen immer im Rudel (Name + Datum + Ort, überall zusammen). → Als ein benanntes Paket bündeln und das Paket weitergeben.
  • Nackte Behelfswerte (primitive obsession): Eine bloße Zahl oder ein Textfeld steht für etwas, das eigentlich ein eigener Begriff ist. → Dem Ding einen eigenen kleinen Namen samt Regeln geben.
  • Wiederholte Weichen (repeated switches): Dieselbe Fallunterscheidung ("wenn Mitglied, dann …") taucht an mehreren Stellen auf. → Einmal entscheiden, überall wiederverwenden.
  • Streu-Operation (shotgun surgery): Eine einzige logische Änderung zwingt zu Anpassungen an zig verstreuten Stellen. → Zusammenlegen, was sich gemeinsam ändert.
  • Vielzweck-Baustelle (divergent change): Ein und dieselbe Datei wird aus lauter verschiedenen Gründen angefasst. → Aufteilen, bis jedes Teil nur noch einen Änderungsgrund hat.
  • Vorrats-Verallgemeinerung (speculative generality): Extras, Schalter und Vorkehrungen "für später", die niemand bestellt hat. → Löschen; erst wieder einbauen, wenn ein echter Bedarf da ist.
  • Befehlsketten (message chains): Um an eine Auskunft zu kommen, muss man sich durch fünf Stationen durchfragen. → Eine direkte Auskunftsstelle einbauen.
  • Durchreicher (middle man): Ein Zwischenglied, das fast nur weiterleitet. → Streichen und direkt beim Ziel fragen.
  • Ausgeschlagenes Erbe (refused bequest): Etwas übernimmt eine Vorlage, ignoriert aber das meiste davon. → Die Verwandtschaft lösen und nur nehmen, was gebraucht wird.
Merksatz bleibt: Fertig ist nur, was einen frischen Beleg hat

Auch ohne eigenen Skill gilt die Regel weiter: Eine "Fertig"-Meldung zählt erst, wenn eine Prüfung gelaufen ist, die auch hätte schiefgehen können — und die KI ehrlich sagt, was sie NICHT geprüft hat. Frag im Zweifel einfach: "Was hast du wirklich geprüft?"

P4.01 Selbst

Partnercheck: Vertrauen entsteht durch Prüfung

Die Prüfung ersetzt deinen Geschmack nicht. Beantworte vier verschiedene Fragen, zu jeder genau einen kurzen Satz: Stärke, Risiko, konkrete Verbesserung und möglicher nächster Schritt. Findest du nach ehrlicher Prüfung kein materielles Risiko oder keine nötige Verbesserung, schreib genau das statt eines erfundenen Mangels. Eine zweite Person liefert beim letzten Punkt nur einen Vorschlag; welchen Schritt du wirklich gehst, entscheidest du. Solo funktioniert derselbe Check mit Abstand oder laut vorgelesen.

Eine Stärke

Was ist gut und soll auf keinen Fall wegrepariert werden?

Ein Risiko

Welche Stelle könnte falsch, erfunden oder missverständlich sein?

Eine Verbesserung

Welche konkrete Änderung macht das Ergebnis besser?

Ein möglicher nächster Schritt

Was wäre jetzt sinnvoll? Du entscheidest anschließend selbst.

P4.02 Selbst

Jetzt du (~12 Min.)

Dein Durchgang am eigenen Projekt
  1. Dein bestes Ergebnis erst selbst benutzen: durchgehen, als würdest du es morgen brauchen.
  2. In einem neuen Chat mit dem Zwei-Achsen-Prompt prüfen: Ergebnisdatei, Ticket, Zieldokument und relevante Projektregeln beifügen.
  3. Materielle Befunde gezielt reparieren lassen: jeweils eine Stelle, eine Änderung, Stärken behalten. Gibt es keinen belegten Befund, erfinde keinen.
  4. Nach jeder Reparatur die gespeicherte Datei erneut öffnen und die betroffene Stelle sowie alle Fertigkriterien noch einmal prüfen. Beleg und Ergebnisversion im Ticket ergänzen; State: closed nur bei vollständiger Erfüllung.
  5. Partnercheck: vier Fragen, vier Sätze — mit einem zweiten Menschen, oder als Selbst-Check mit Abstand.
Phase fertig, wenn …
  • jedes eingesammelte Ergebnis ein Prüfurteil hat: nutzbar / reparieren / selbst klären,
  • Ungeprüftes als Platzhalter markiert ist statt echt auszusehen,
  • du aus Partnercheck oder Review einen konkreten nächsten Schritt notiert hast.

Phase 5 · Die Schleife · ~13 Min.

Die Schlussrunden: Neu ist hier nichts — neu ist nur, dass es kreist, bis DU aufhörst

Zweck: "Fertig" ist keine Meldung der KI, sondern deine begründete Entscheidung. Du benutzt das Ergebnis selbst; ein frischer Chat prüft zusätzlich, was der Bau-Chat übersehen haben könnte. Nur materielle Funde werden Tickets. Abschlussreif ist das Projekt erst, wenn keine notwendige offene Arbeit mehr übrig ist oder verbleibende Punkte ausdrücklich mit Grund vertagt beziehungsweise aus dem Ziel ausgeschlossen sind. Für den ersten Durchgang reicht: eine ehrliche kleine Runde und die wichtigste belegte Korrektur, falls es eine gibt.

Werkstatt-Prinzip

"Die erste Fassung ist der Anfang der Schlussrunde, nicht das Ende des Projekts."

Hier kommt alles aus den Phasen 1 bis 4 noch einmal im Kleinen zusammen: Du suchst nach materiellen Mängeln, machst echte Funde zu prüfbaren Tickets, lässt sie in frischen Chats umsetzen und prüfst erneut. Die Runden enden, wenn nichts Notwendiges mehr offen ist; eine feste Zahl gibt es nicht.

Zwei Prüfer, eine Schleife

Das ist dieselbe Bewegung wie in Phase 4, nur dass die ersten Funde jetzt von dir kommen statt vom Agenten: Dein Review (Schritte 1–2) findet, was echte Nutzer stört; das sieht keine KI für dich. Das Agenten-Review (Schritt 4) findet im frischen Kontext, was der Bau-Chat übersehen hat. Beide münden in dieselbe Ablage und denselben Umsetzungs-Schritt. Und dein Zufrieden ist das einzige Tor nach draußen: Kein Skill beendet diese Schleife, das tust du.

Schritt für Schritt (~6 Min.)
  1. Nimm das Ergebnis aus Phase 4 und benutz es wie ein echter Gast oder Nutzer: Was stört? Was fehlt?
  2. Sortiere die Funde im Chat: Was ist ein echtes Ticket, was ist Kleinkram für nebenbei?
  3. Nimm nur materielle Funde als neue Tickets in die Ablage auf, mit Fertig-Kriterien und, wo es hilft, einem Screenshot. Erfinde keine Mindestzahl.
  4. Gibt es einen notwendigen Fund, setz den wichtigsten um: frischer Chat, lokaler Einzelticket-Auftrag plus Ticketdatei. Gibt es keinen, dokumentiere kurz die durchgeführte Prüfung.
  5. Lass in einem weiteren frischen Chat mit dem Zwei-Achsen-Prompt prüfen. Besser? Neue Baustellen?
  6. Nimm auch aus dem Agenten-Review nur belegte, materielle Funde als Tickets auf und starte für das wichtigste bei Bedarf den Einzelticket-Auftrag.
  7. Entscheide bewusst: noch notwendige offene Arbeit oder abschlussreif? Abschlussreif nur, wenn kein notwendiges Ticket State: open trägt; Vertagtes und Nicht-Ziele brauchen einen sichtbaren Grund.
P5.01 Selbst

Jetzt du (~7 Min.)

Dein Durchgang am eigenen Projekt
  1. Benutze dein bestes Ticket-Ergebnis zwei Minuten lang wie dein späterer Nutzer. Notiere jeden materiellen Fund: was stört, was fehlt, was ist unklar. Findest du keinen, dokumentiere die Prüfung statt einen Mangel zu erfinden.
  2. Nimm echte Funde als neue Tickets in die Ablage auf, jeweils mit prüfbarem Fertig-Kriterium und bei Bedarf Screenshot.
  3. Gibt es ein notwendiges Ticket, setz das wichtigste in einem frischen Chat mit dem Einzelticket-Auftrag um und öffne die gespeicherte Ergebnisdatei erneut.
  4. Wenn die Zeit reicht: frischer Zwei-Achsen-Review; dessen Funde ebenfalls als Tickets aufnehmen. Danach bewusst entscheiden: noch eine Runde oder zufrieden?
Phase fertig, wenn …
  • alle materiellen Funde als Tickets mit Fertig-Kriterien erfasst sind oder die Prüfung ehrlich „kein materieller Befund“ ergeben hat,
  • notwendige Tickets umgesetzt und erneut geprüft sind oder mit sichtbarem Grund vertagt beziehungsweise außerhalb des Ziels liegen,
  • du weißt, dass Review-Funde des Agenten denselben Weg gehen: Ticket, frischer Chat, umsetzen,
  • du bewusst entschieden hast: weiterarbeiten oder abschlussreif; abschlussreif heißt, dass kein notwendiges Ticket mehr State: open trägt.

Phase 6 · Abschließen · ~6 Min.

Zufrieden? Dann schließ dein Projekt so ab, wie es zu ihm passt

Zweck: Der Abschluss ist ein eigener, bewusster Schritt, und er sieht bei jedem Projekt anders aus. Veröffentlichen, übergeben, ablegen oder sauber pausieren: Du entscheidest, was „zu“ bedeutet, klärst die Restliste und hältst Ergebnis, Prüfstand, vertagte Punkte und nächsten Schritt im Abschlussbericht fest. Für den ersten Durchgang reicht: Abschluss-Form gewählt und Abschlussbericht gespeichert; ein Handoff kommt nur dazu, wenn Arbeit oder Kontext wirklich übergeben wird.

Werkstatt-Prinzip

"Ein Projekt ist nicht fertig, wenn nichts mehr zu tun ist — sondern wenn du entschieden hast, was mit ihm passiert."

Was der Abschluss konkret ist, weißt du oft erst, wenn du an diesem Punkt stehst: Die Vereinsfeier-Planung geht an den Vorstand, die kleine Webseite geht online, die Recherche wandert sauber benannt in den Ordner, und manche Projekte enden nie, sie werden nur bis zur nächsten Saison geparkt. Alles davon ist ein guter Abschluss, solange er bewusst passiert.
Veröffentlichen

Etwas soll online oder raus an Empfänger: "Hilf mir, das Projekt zu veröffentlichen (zu deployen)." Die KI schreibt die Schritte, du gehst sie durch; Versand und Freischalten bleiben Mensch-Entscheidungen.

Übergeben

Jemand anderes übernimmt: /handoff schreibt die Übergabe, dazu Zieldokument und Entscheidungen aus dem Projektordner. Wer übernimmt, startet in der Smart Zone statt bei null.

Archivieren

Das Ergebnis ist im Einsatz, das Projekt ist durch: Erledigtes abhaken, Ergebnis sauber benennen und ablegen, Ordner zu. Auffindbar bleibt alles, aber nichts liegt mehr offen herum.

Pausieren

Das Projekt endet nicht, es ruht: /handoff für dein Zukunfts-Ich, offene Tickets bleiben in der Ablage, Wiedervorlage in den Kalender. Die nächste Sitzung startet frisch, mit Notizen.

Ergebnis der roten Abschlusskarte · Abschlussbericht
Lies Zieldokument, lokale Tickets samt Comments und State, Ergebnisdateien sowie Prüfnotizen im Projektordner. Erstelle daraus einen knappen Abschlussbericht:
1. Ziel und Abschluss-Form: veröffentlichen, übergeben, archivieren oder pausieren
2. Ergebnis: Was liegt jetzt nutzbar vor, mit Dateinamen und Version?
3. Geprüfter Stand: Welche Prüfungen und Belege gibt es?
4. Offen oder bewusst vertagt: Was ist nicht fertig, warum, und gehört es noch zum Ziel?
5. Nächster realer Schritt: Wer tut was bis wann?

Behaupte keine Erledigung ohne Datei und Beleg. Wenn noch ein notwendiges Ticket State: open trägt, nenne das Projekt nicht abgeschlossen, sondern „pausiert“ oder „noch in Arbeit“. Veröffentliche, versende, lösche oder verschiebe nichts.

Speichere den Bericht als Markdown-Datei im Projektordner unter „abschlussbericht-YYYY-MM-DD.md“. Öffne die gespeicherte Datei erneut, prüfe die fünf Abschnitte und nenne mir anschließend den genauen Pfad.

Unterscheide den Bericht von einem Handoff: Der Abschlussbericht bilanziert das Projekt. Ein Handoff übergibt dagegen den Arbeitskontext an eine neue Sitzung oder Person.
Die Abschlussfragen

Ganz zum Schluss darf die KI noch einmal planen helfen. Frag sie: "Was ist alles noch zu tun, damit wir das Projekt abschließen können?" Die Antwort ist deine letzte kleine Ticket-Liste. Erst wenn sie leer oder bewusst vertagt ist, ist das Projekt zu, auf die Art, die du oben gewählt hast.

Im Original bei Matt · Doc Rot

Matt lässt erledigte Zieldokumente nicht offen herumliegen. Sein Grund: Einen Monat später hat sich das Projekt weiterbewegt; findet die KI dann das alte Dokument, hält sie es für die Wahrheit und arbeitet in die falsche Richtung. Deshalb gehört zu jedem Abschluss: Erledigtes schließen oder abhaken; auffindbar bleibt es trotzdem.

Schritt für Schritt (~3 Min.)
  1. Stell im Projekt-Chat die Abschlussfrage: "Was ist alles noch zu tun, damit wir das Projekt abschließen können?"
  2. Geh die Antwort durch: Was davon wird noch ein Ticket, was wird bewusst vertagt?
  3. Wähle die Abschluss-Form, hak Erledigtes mit Beleg ab und kopiere den Abschlussbericht-Auftrag. Öffne danach die gespeicherte Datei und notiere ihren Pfad. Nur bei Übergabe oder Pause erzeugst du zusätzlich ein Handoff für die nächste Sitzung oder Person.
P6.01 Selbst

Jetzt du (~3 Min.)

Deinen Abschluss festlegen
  1. Entscheide für dein Projekt: veröffentlichen, übergeben, archivieren oder pausieren? Nach dem ersten Durchgang ist "pausieren" für die meisten die ehrliche Antwort.
  2. Stell die Abschlussfrage im Chat, ordne die Restliste und lass danach den Abschlussbericht als Datei im Projektordner erstellen.
  3. Nur wenn jemand oder eine spätere Sitzung weiterarbeitet: zusätzlich /handoff nutzen. Erledigtes abhaken und den nächsten realen Schritt mit Datum notieren.
Phase fertig, wenn …
  • du weißt, welche Abschluss-Form zu deinem Projekt passt,
  • die Restliste aus der Abschlussfrage als Tickets in der Ablage liegt oder bewusst vertagt ist,
  • der gespeicherte Abschlussbericht Ergebnis, Prüfstand, offene Punkte und nächsten Schritt festhält und sein Pfad bekannt ist,
  • Erledigtes abgehakt ist und ein Handoff nur dann vorliegt, wenn tatsächlich eine Übergabe nötig ist.

Der Fahrplan ohne Agenten-Tool (Weg B)

Dieselben 8 Schritte, nur führst du die Aufgabentafel selbst als normales Dokument oder Schreib-Block und überträgst die benötigten Texte zwischen deinen Chats:

  1. Aufgabentafel-Dokument anlegen statt Werkstatt einrichten.
  2. Denken ohne KI: Steckbrief auf Papier, unverändert.
  3. Grillen: den deutschen grilling-Text von der Skill-Karte als Prompt einfügen.
  4. Zieldokument und Tickets im Chat formulieren lassen, beides in dein Dokument kopieren.
  5. Pro Ticket ein neuer Chat: Ticket + Zieldokument + Einzelticket-Auftrag einfügen, Ergebnis als Datei sichern, Belege und Status ins Dokument zurücktragen.
  6. Prüfen: Ergebnis selbst öffnen; im frischen Chat den Zwei-Achsen-Prüfauftrag aus Phase 4 nutzen: Qualität und Auftrag strikt getrennt.
  7. Funde als neue Tickets auf die Tafel, umsetzen wie in 5.
  8. Abschließen: Abschlussbericht-Auftrag aus Phase 6 nutzen und die Tafel aufräumen.

Du wirst das Abtippen satt: Genau dafür gibt es Skills.

Egal, wo dein Projekt steht

Der Fahrplan hat keinen Zwang zur Reihenfolge ab Schritt 5: Steckst du mitten in einem laufenden Projekt, steig dort ein, wo es hakt. Unklares Ziel? Schritt 3. Chaos im Ordner? Werkzeugkoffer. Ergebnisse ungeprüft? Schritt 6. Erste Fassung steht? Schritt 7. Zufrieden? Schritt 8.

T4.01 Selbst

Abschlusscheck: Was tust du, wenn …?

… die KI "fertig!" meldet und du ihr nicht glaubst?

Beleg verlangen aus einer Prüfung, die auch hätte fehlschlagen können — "Was hast du wirklich geprüft?". Merksatz: Fertig ist nur, was einen frischen Beleg hat. (Als fester Skill: /verify-before-done aus dem Zusatzpaket.)

… dein Projekt-Chat zäh wird und Regeln übersieht?

Überladener Kontext ist eine mögliche Ursache: Ergebnis und Handoff prüfen, Auftrag fokussieren und im frischen Chat gegenproben.

… eine Aufgabe zu groß wirkt, um anzufangen?

Richtig groß und neblig? Erst /wayfinder: Landkarte zeichnen, Ticket für Ticket klären. Sonst /grill-with-docs (verstehen), dann /to-spec und /to-tickets: schneiden, bis jedes Ticket in einen kurzen Chat passt.

… ein wichtiges Dokument raus soll und du unsicher bist?

Ergebnis selbst öffnen und im frischen Chat mit dem Zwei-Achsen-Auftrag prüfen: Qualität und Auftrag getrennt. Danach den Vier-Fragen-Partnercheck mit einem Menschen machen.

… dein Projekt eine Geld-, Rechts- oder Gesundheitsfrage berührt?

KI für Vorbereitung und Fragenliste nutzen, entschieden wird mit Fachleuten. Und wie immer: Thema ja, Personalien nein.

… du denselben Ablauf zum dritten Mal von Hand machst?

/skill-creator: den Ablauf beschreiben, SKILL.md erzeugen lassen, in die Ablage legen, ab jetzt ist es ein Werkzeug.

… du mit dem Ergebnis zufrieden bist, und dann?

Abschlussfrage stellen ("Was fehlt noch zum Abschluss?"), dann bewusst wählen: veröffentlichen, übergeben, archivieren oder pausieren. Erledigtes abhaken, damit nichts fault.
